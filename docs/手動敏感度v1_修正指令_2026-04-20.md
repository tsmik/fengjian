## Bug A-1：資料完整度檢查（最重要）

**問題：** 當手動資料不完整（有 null 格）時，仍然做診斷和 Top N 計算，造成所有數字都是基於殘缺資料算出的誤判。

**修正要求：**

1. **新增完整度判斷函式**

   在 `js/manual_sens_v2.js` 開頭工具層區，新增：

   ```javascript
   // 檢查指定維度範圍的所有部位是否填完
   // dimIndices: 維度編號陣列（例如老闆是 [0,1,2]）
   // 回傳 {complete: boolean, filled: number, total: number}
   function checkBlockComplete(data, dimIndices) {
     var total = dimIndices.length * 9;
     var filled = 0;
     dimIndices.forEach(function(di) {
       for (var pi = 0; pi < 9; pi++) {
         if (data[di][pi] === 'A' || data[di][pi] === 'B') filled++;
       }
     });
     return { complete: filled === total, filled: filled, total: total };
   }
   ```

2. **各區塊渲染前檢查**

   `renderBossCard`、`renderManagerCard`、`renderLuckBlock`、`renderAcquiredBlock` 四個函式在計算和渲染診斷前，都要先呼叫 `checkBlockComplete`。

   對應區塊範圍：
   - 老闆小卡：`[0, 1, 2]`
   - 主管小卡：`[3, 4, 5]`
   - 運氣區塊：`[6, 7, 8]`
   - 後天區塊：`[9, 10, 11, 12]`

   若 `complete === false`，該區塊**不做診斷**，改為顯示：

   ```
   ⚠ [區塊名] 尚未填完（已填 X/Y 格）
   請先到「手動輸入兵法報告」填完整所有維度和部位，再回來此頁查看診斷。
   ```

3. **角色定位也要檢查**

   「老闆 vs 主管」角色定位需要先天 6 維度全填完才能判定，否則顯示「資料尚未填完，暫無法判定角色定位」。

4. **Top N 各區塊也要檢查**

   `renderTopNList` 在渲染各區塊 Top N 前，同樣先檢查該區塊完整度。
   - 若不完整，該區塊的 Top N 顯示「⚠ [區塊名] 尚未填完，暫無翻轉建議」
   - 先天 Top 3 的完整度 = 老闆完整 AND 主管完整
   - 不完整時不做翻轉計算（避免浪費運算）

5. **頁面總入口檢查**

   在 `renderManualSensV2Page` 開頭加總檢查：若整份資料完全是空的（全 null），顯示頂部提示：

   ```
   ⚠ 手動輸入報告尚未填寫
   [按鈕：前往手動輸入報告]
   ```

   其他情況正常進入各區塊（各區塊內自己處理完整度）。

## Bug A-2：均衡度柱子圖沒有視覺化

**問題：** spec 說要有 bar chart 呈現靜側柱子，實際上截圖看到的只有數字和文字。

**修正要求：**

在 `renderBossCard` 和 `renderManagerCard` 的均衡度區塊，把目前只顯示數字的排版改為**真的柱子圖**：

```
均衡度（靜側柱子圖）：

形 ████████ 5
經 ████████████ 7
方 ██████ 3

差距 4  ⚠ 需提醒短板
短板：方 — 成就方向不明
```

**規格：**
- 用 HTML div 做橫向柱子，寬度依靜側數字比例（靜側 9 格滿分 = 最長）
- 柱子顏色：與該係數標題色一致（老闆用蘇芳 `#8E4B50`，主管用丁子茶 `#8C6B4A`）
- 柱子高度：14px 左右
- 柱子右邊顯示數字
- 柱子左邊標籤：形/經/方 或 直/收/緩

先天 Top 3 不需要自己的柱子圖（它沒有獨立的「先天小卡」），老闆主管兩張卡各自顯示自己的即可。

## Bug A-3：`type` 動靜判定反了

**問題：** 截圖看到「分合 0.13，靜」和「順逆 0.29，靜」——係數 0.13/0.29 的情況下應該是「動」，不是「靜」。

**修正要求：**

在 `renderLuckBlock` 和 `renderAcquiredBlock` 顯示維度動靜時，不要用 `calcDim(data, di).type`，直接檢查靜側柱子數量與動側柱子數量的比較：

```javascript
// 正確判斷動靜方向
var dimRes = calcDim(data, di);
if (!dimRes) { /* 不完整 */ return; }

// 靜側數字 vs 動側數字（從 calcDim 的 a/b 欄位讀，a 是靜側時才是靜）
// 看 core.js 裡 DIMS[di] 的 a/b 對照，確認何為靜側
var staticCount = dimRes.a; // 依 DIMS 設定，靜側是 a 或 b
var dynamicCount = dimRes.b;
// ...取得正確的 type
```

**實作前**，先去 `js/core.js` 看 `calcDim` 如何決定 `type`，可能現有函式本身就有 bug。如果是 `calcDim` 本身有問題，修 `calcDim`；如果是 `manual_sens_v2.js` 用錯欄位，修 `manual_sens_v2.js`。

**驗證方式：** 改完後跑一次，係數 0.13 的維度應該顯示「動」，係數 0.80 的維度應該顯示「靜」（當靜側是多數時）。

## Bug B-1：主管係數方向判讀話術不完整

**問題：** 目前「直>曲 ✗」單獨一項就觸發「幕僚/幕僚長傾向」，但 GS_15 §3 老師原話是「**全部曲沒有直**，是幕僚/幕僚長的概念」——要「全部曲」才算。

**修正要求：**

在 `calcMgrDirection` 函式，把方向判讀話術改為以下規則（按優先順序判斷）：

```javascript
// 取得主管係數方向判讀話術
function calcMgrDirection(data) {
  var qu = calcDim(data, 3);   // 曲直
  var shou = calcDim(data, 4); // 收放
  var huan = calcDim(data, 5); // 緩急

  if (!qu || !shou || !huan) return null; // 不完整不判

  // 規則 1：全部曲沒有直 → 幕僚長概念（老師原話）
  // 條件：曲直極偏曲（曲=9 或 直=0-1）
  if (qu.b === 9 || qu.a <= 1) {
    return {
      text: '全部曲沒有直，幕僚/幕僚長概念（老師原話）',
      signs: [{ label: '直>曲', ok: false }, ...]
    };
  }

  // 規則 2：典型穩當主管（GS_15 §4）
  // 條件：直>曲、收>放、緩>急，三個靜側都佔多數
  if (qu.a > qu.b && shou.a > shou.b && huan.a > huan.b) {
    return {
      text: '典型穩當主管 — 擔當、能耐、節奏一致',
      signs: [{ label: '直>曲', ok: true }, { label: '收>放', ok: true }, { label: '緩>急', ok: true }]
    };
  }

  // 規則 3：部分方向不符
  return {
    text: '方向不完全符合，看係數 0.80 是調整機會',
    signs: [
      { label: '直>曲', ok: qu.a > qu.b },
      { label: '收>放', ok: shou.a > shou.b },
      { label: '緩>急', ok: huan.a > huan.b }
    ]
  };
}
```

老闆 `calcBossDirection` 邏輯類似，但老闆沒有「全部陰沒有陽」這種老師原話特例，不需要規則 1。
